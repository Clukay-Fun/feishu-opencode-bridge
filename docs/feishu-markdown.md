# 飞书输出 Markdown 语法规范

> 适用于 `feishu-opencode-bridge` 所有卡片 body 和 Plain Post 的 Markdown 内容。

## 一、支持的语法

### 文字样式

| 场景       | 语法           | 示例                               | 备注           |
| ---------- | -------------- | ---------------------------------- | -------------- |
| 强调关键词 | `**粗体**`     | **已切换**                         |                |
| 命令名     | `` `反引号` `` | `/status`                          |                |
| 代码片段   | `` `反引号` `` | `rm -rf dist/`                     |                |
| 代码块     | ` ``` ` 包裹   | 多行命令                           | 不加语言标注 |
| 链接       | `[文字](url)`  | [opencode.ai](https://opencode.ai) |                |
| 已失效内容 | `~~删除线~~`   | ~~旧会话~~                         |                |

### 禁用样式

| 语法       | 原因           |
| ---------- | -------------- |
| `*斜体*`   | 飞书渲染不稳定 |
| `_下划线_` | 飞书渲染不稳定 |

### 标题

```markdown
### 三级标题（卡片内分段标准）
#### 四级标题（极少用，仅嵌套分段）
```

- 卡片内分段统一用 `###`，不用 `#` 或 `##`（飞书渲染字号过大）
- `Notice Card` 不用标题
- 每个 `###` 前面必须有一个空行

### 列表

有序列表只用于步骤或编号选择：

```markdown
1. 先做这个
2. 再做这个
```

无序列表用于并列项，统一用 `-`，不用 `*`：

```markdown
- 项目 A
- 项目 B
```

会话列表固定格式：

```text
#1  会话名称  ← 当前
#2  会话名称  04-06 15:10
```

嵌套列表最多一层，不要更深。

### 代码

行内代码：`` `代码` ``

代码块不带语言标注：

````markdown
```
npm run dev
```
````

飞书可能把 ` ```bash `、` ```sh ` 等语言标注错误渲染为正文前缀。
formatter 会在发送前移除 fenced code block 的语言标注。

### 链接

统一用 `[显示文字](url)`，不裸贴 URL。

### 引用

用于原始命令回显或上下文补充说明（飞书渲染为灰色背景条）：

```markdown
> 引用内容
```

### 表格

用于 `/models` 等多字段列表：

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| A   | B   | C   |
```

- 不支持对齐语法（`:---:` / `---:` 无效），统一左对齐
- 列数不宜超过 4 列，移动端显示受限

### 分隔线

用 `---` 在逻辑段落之间拉开视觉间距。适用于：

- `Command Card` 正文与底部操作提示之间
- `Process Card` 长内容的逻辑分段处（超过 10 行插一次）
- `Notice Card` 不用分隔线

## 二、飞书不支持的语法

以下语法在飞书卡片 Markdown 中不渲染或行为异常，一律禁止：

| 语法                    | 说明                                      |
| ----------------------- | ----------------------------------------- |
| HTML 标签               | `<details>`、`<summary>`、`<br>` 等被忽略 |
| 图片 `![alt](url)`      | 卡片 Markdown 不渲染，需用 `img` 组件     |
| 脚注 `[^1]`             | 不支持                                    |
| 任务列表 `- [ ]`        | 不支持                                    |
| 表格对齐 `:---:`        | 不支持，只有左对齐                        |
| `*斜体*` / `_下划线_`   | 渲染不稳定，禁用                          |
| `#` 一级标题 / `##` 二级 | 卡片内字号过大，禁用（统一用 `###`）      |

## 三、Emoji 使用规范

> Emoji 是状态指示器，不是装饰。

### 规则

#### 1. 一对一绑定

一个语义只绑一个 emoji，一个 emoji 只表达一个语义。

#### 2. 只出现在卡片 body / subtitle 的文本内容里

- `Notice Card` body 行首第一个字符
- `Process Card` subtitle
- `Command Card` body 内的状态标记

不出现在：

- 卡片 header title（header 已有 `standard_icon` + `template` 颜色承担语义）
- 正文中间（不在句中插入 emoji）
- `Plain Post`（除非降级兜底需要最低限度的状态提示）

#### 3. 映射表穷举，表外不用

| Emoji | 唯一语义      | 典型出现位置                       |
| ----- | ------------- | ---------------------------------- |
| ⚙️    | 处理中        | Process Card subtitle              |
| ✅    | 成功 / 完成   | Process Card subtitle, Notice body |
| ❌    | 错误 / 失败   | Notice body, Process Card subtitle |
| ⚠️    | 警告 / 提醒   | Notice body                        |
| ℹ️    | 信息 / 空状态 | Notice body                        |
| 🔐    | 权限请求      | Notice body                        |
| ⏹     | 中止          | Notice body                        |
| ⏱     | 耗时          | Process Card subtitle              |
| ⏳    | 队列等待      | Process Card subtitle              |

不使用 🎉 🚀 👋 💬 👤 🚪 🤖 📋 等，保持运维面板调性。

## 四、长度控制

| 位置              | 限制                     | 处理                                        |
| ----------------- | ------------------------ | ------------------------------------------- |
| Notice Card body  | 1-2 行                   | 超出则考虑升级为 Command Card               |
| Command Card body | 无硬限制                 | 内容过长时用 `###` 分段                     |
| Process Card body | 单条卡片上限 30,000 字符 | 超出时截断，末尾追加 `⚠️ 输出过长，已截断` |
| Plain Post        | 极短                     | 超过 3 行应考虑是否该用卡片                 |

## 五、转义规范

| 字符    | 处理方式                               |
| ------- | -------------------------------------- |
| `<` `>` | 包在反引号内：`` `<provider/model>` `` |
| `&`     | 代码上下文中用反引号包裹               |
| `\|`    | 表格内需转义为 `\|`                    |
| `` ` `` | 嵌套时用双反引号                       |

原则：凡是出现在代码、命令、路径中的特殊字符，一律用反引号包裹。

**重要**：代码块（` ``` ` 包裹）内的内容不做 HTML escape。`->` 不应被转成 `-&gt;`。formatter 构建卡片时需确保 fenced code block 内容原样传入。

## 六、空行与间距

飞书卡片 Markdown 的行间距由平台渲染决定，不可自定义行高或字间距。可用的间距控制手段：

| 手段                 | 效果                   | 用法                         |
| -------------------- | ---------------------- | ---------------------------- |
| 空行                 | 段落间距               | 段落间一个空行，不多不少     |
| `---` 分隔线         | 视觉断开，比空行更明显 | 逻辑分段处、正文与 footer 间 |
| `###` 标题前空行     | 拉开与上文的距离       | 每个 `###` 前必须有空行      |
| 代码块前后空行       | 拉开代码和正文的距离   | 代码块前后各留一个空行       |
| 引用 `>`             | 灰色背景条，自带视觉分隔 | 补充说明、命令回显         |
| `notation` 字号      | 卡片组件级小字         | footer 提示文字              |
| 多 markdown 组件拆分 | 用卡片 JSON 的 vertical_spacing | 密集内容需要更大间距时 |

具体规则：

- 段落间一个空行，连续空行可能被合并或产生异常间距
- 列表项之间不加空行（加了会变成松散列表，间距过大）
- 代码块前后各留一个空行
- 长内容（超过 10 行）在逻辑分段处插 `---`

## 七、语气规范

> 运维面板调性：克制、准确、不废话。

### 句式规则

| 规则       | 句式               | 示例                       |
| ---------- | ------------------ | -------------------------- |
| 完成态     | `已` 开头          | 已切换、已解除绑定、已中止 |
| 进行态     | `正在` 开头        | 正在执行、正在创建会话     |
| 状态态     | `当前` 开头        | 当前无活跃会话             |
| 引导下一步 | `发送 /xxx + 动词` | 发送 `/abort` 中止任务     |

### 禁止用法

| 禁止       | 正确写法                                                     |
| ---------- | ------------------------------------------------------------ |
| 感叹号     | ✅ 已切换到会话 #2 ❌ 已切换到会话 #2！                      |
| "抱歉"     | ✅ 服务不可用，请检查连接 ❌ 抱歉，服务不可用                |
| "请注意"   | ✅ 直接说明情况 ❌ 请注意，会话已过期                         |
| 解释原因   | ✅ 当前没有正在执行的任务 ❌ 因为没有任务在执行，所以无法中止 |
| "可以"     | ✅ 发送 `/sessions` 查看列表 ❌ 发送 `/sessions` 可以查看列表 |
| 裸写命令名 | ✅ 发送 `/sessions` ❌ 发送 /sessions                        |

## 八、各卡片类型的 Markdown 用法速查

### Command Card

```markdown
### 会话列表

#1  重构登录模块  ← 当前
#2  修复 CI 流水线  04-06 15:10
#3  API 文档更新  04-05 09:30

---

发送 `/switch <编号>` 切换会话
```

### Notice Card

```markdown
ℹ️ 当前没有待确认的权限请求。
```

```markdown
⚠️ 会话列表已过期，请重新发送 `/sessions`
```

```markdown
❌ OpenCode 服务不可用，请检查连接。
```

不用标题、不用分隔线、1-2 行结束。

### Plain Post

```text
Session abc123 created.
Model: claude-sonnet-4-20250514
```

只用于 passthrough 文本输出、极短确认语、卡片不可用时的降级兜底。bridge 自有命令、结构化列表、系统提示一律不使用 Plain Post。

### Process Card

正常对话输出，遵循上述所有语法规范。

长文本输出补充规则：

- 用 `###` 切分段，每段不超过一屏
- 每个 `###` 前加空行，长段落间插 `---`
- 调用链、时序、数据流用 fenced code block，不用自然语言描述
- 文件路径不散落在正文句子里，集中放段末或单独成块
- 行号不写进正文
- 解释用短句，调用顺序用结构，两者不混写

推荐示例：

````markdown
### 启动时序

main() 依次执行：

1. loadConfig()
2. runStartupPreflight()
3. BridgeApp.start()
4. HTTP server
5. Feishu WebSocket

---

### 消息处理

```text
飞书事件 → ws.ts handleEvent()
  → 去重 / 白名单 / 消息类型校验
  → app.handleIncomingMessage()
    → router 判定命令 or 对话
    → runTurn()
      → 确保 session 存在
      → 发送状态卡
      → prompt_async + SSE 增量
      → 回写飞书卡片
```
````
