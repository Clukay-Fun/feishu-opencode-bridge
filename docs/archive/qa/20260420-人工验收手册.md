# 4/20 Acceptance Runbook

## 使用方式

这份 runbook 用于 4/20 提交版的最终人工验收。每一项都包含：

- 操作
- 预期结果
- 证据

执行时建议同时保留：

- 飞书截图或短 GIF
- bridge 控制台日志
- 如有部署环境，再补一份服务器终端输出

## A. Local Automated Checks

### A1. 安装与构建

1. 执行 `npm ci`
   预期：依赖安装成功，退出码为 `0`
   证据：终端输出截图

2. 执行 `npm run build`
   预期：TypeScript 构建成功，退出码为 `0`
   证据：终端输出截图

3. 执行 `npm run typecheck`
   预期：类型检查通过
   证据：终端输出截图

4. 执行 `npm run lint`
   预期：Lint 通过
   证据：终端输出截图

5. 执行 `npm test`
   预期：Vitest 全量通过
   证据：终端输出截图

### A2. 启动前 preflight

1. 执行 `npm run dev:once`
   预期：启动前 preflight 通过，bridge 进入稳定运行态
   证据：终端输出截图

## B. Private Chat Checks

### B1. 新建与当前状态

1. 在飞书私聊发送 `/new`
   预期：收到“已创建新会话” Command Card
   证据：卡片截图

2. 发送一条普通开发任务
   预期：出现 Process Card，状态会从处理中更新到完成
   证据：处理中与完成态截图各一张

3. 发送 `/status`
   预期：收到“会话状态” Command Card，包含当前会话和系统状态
   证据：卡片截图

### B2. 会话列表与切换

1. 发送 `/sessions`
   预期：收到“会话列表” Command Card，当前会话有 `当前` 标记
   证据：卡片截图

2. 再执行一次 `/new`
   预期：创建第二个会话，收到“已创建新会话” Command Card
   证据：卡片截图

3. 再次发送 `/sessions`
   预期：列表里至少有两个会话
   证据：卡片截图

4. 发送 `/switch 1`
   预期：收到“已切换会话” Command Card，显示离开旧会话并进入目标会话
   证据：卡片截图

5. 发送 `/switch 999`
   预期：收到无效编号提示
   证据：提示截图

### B3. 模型查看与切换

1. 发送 `/model`
   预期：收到“可用模型”卡片，每个 provider 展示最近 5 个模型
   证据：卡片截图

2. 发送 `/model openai`
   预期：只显示 OpenAI 的模型列表
   证据：卡片截图

3. 发送 `/model use openai/gpt-5.4-mini`
   预期：OpenCode 接收模型切换命令，后续任务继续正常执行
   证据：回复截图

4. 发送 `/model reset`
   预期：恢复默认模型
   证据：回复截图

### B4. 中止

1. 在空闲窗口发送 `/abort`
   预期：收到灰色 Notice Card，正文为“当前没有正在执行的任务。”
   证据：卡片截图

2. 触发一个明显会持续几秒的任务，在处理中发送 `/abort`
   预期：收到橙色 Notice Card，正文为“当前任务已中止，可发送新消息继续对话。”
   证据：卡片截图

## C. Group Chat Checks

### C1. 绑定与免 @

1. 在群里发送 `@bot` + 普通消息
   预期：机器人正常响应，当前用户完成绑定
   证据：群消息截图

2. 不再 `@bot`，直接发送第二条普通消息
   预期：机器人继续响应
   证据：群消息截图

3. 用另一个未绑定账号在群里直接发送普通消息
   预期：bridge 静默忽略，不回复
   证据：群消息截图 + 日志片段

### C2. 群聊命令

1. 已绑定用户发送 `/who`
   预期：收到“群聊绑定状态”卡片，显示绑定人数和“你已绑定”
   证据：卡片截图

2. 已绑定用户发送 `/leave`
   预期：收到“已解除绑定”卡片
   证据：卡片截图

3. `/leave` 后再次直接发送不带 `@bot` 的普通消息
   预期：bridge 静默忽略，不回复
   证据：群消息截图 + 日志片段

4. 未绑定用户发送 `@bot /who`
   预期：可查询状态，但不会顺手绑定
   证据：卡片截图

### C3. 话题窗口

1. 在同一个群内创建两个不同话题
2. 分别在两个话题里触发一次会话
   预期：两个话题各自保持独立 `sessionId`
   证据：日志片段或调试输出

## D. Permission Card Checks

### D1. 真按钮路径

1. 发送一个会触发权限确认的任务
   预期：出现紫色“权限请求”卡片
   证据：卡片截图

2. 点击 `/allow once · 仅此一次`
   预期：原卡直接更新为终态，显示“当前权限请求已确认，可继续执行。”
   证据：点击前后两张截图

3. 重新触发一次权限请求，点击 `/allow always · 始终允许`
   预期：原卡直接更新为终态，显示“后续同类权限将自动允许”
   证据：点击前后两张截图

4. 再触发一次权限请求，点击 `/deny · 拒绝`
   预期：原卡直接更新为终态，显示“当前权限请求已拒绝。”
   证据：点击前后两张截图

### D2. 边界情况

1. 触发权限请求后等待超时，再点击旧卡
   预期：返回“权限请求已超时，已默认拒绝。”
   证据：截图

2. 权限已处理后再次点击旧卡
   预期：返回对应终态，不出现裸错误
   证据：截图

3. 群聊里由非发起者点击按钮
   预期：返回“当前按钮仅限本轮发起者处理。”
   证据：截图

### D3. 文本 fallback

1. 重新触发权限请求
2. 依次验证：
   - `/allow once`
   - `/allow always`
   - `/deny`
   预期：即使不点按钮，文本命令也能完成权限处理
   证据：截图

## E. Deployment Checks

### E1. 本机健康检查

1. 启动 `opencode serve`
   预期：服务正常监听
   证据：终端截图

2. 启动 bridge
   预期：preflight 通过
   证据：终端截图

3. 执行 `curl http://127.0.0.1:3000/healthz`
   预期：返回 `{"ok":true}`
   证据：终端截图

### E2. 域名与回调

1. 通过 Caddy 暴露域名
2. 执行 `curl https://<domain>/healthz`
   预期：返回 `{"ok":true}`
   证据：终端截图

3. 在飞书开放平台配置卡片回调为 `https://<domain>/webhook/card`
   预期：按钮点击能够命中回调并成功更新原卡
   证据：开放平台配置截图 + 飞书卡片截图

### E3. Linux x64 验收

1. 在目标服务器执行：
   - `npm ci`
   - `npm run build`
   - `npm test`
   预期：全部通过
   证据：终端截图

2. 如后续重新引入任何原生依赖
   预期：必须再次在 Linux x64 真机完成上述三步
   证据：终端截图

## F. Recording Checklist

最少保留以下素材：

- 私聊 `/new` + Process Card 完成态
- `/sessions` 会话列表卡
- `/model` 模型卡
- 群聊首次 `@bot` 绑定截图
- `/who` 与 `/leave` 卡片
- 权限请求卡点击前后截图
- `healthz` 成功截图

建议额外录制 3 段短 GIF：

1. Process Card 从处理中到完成
2. 权限按钮点击并更新原卡
3. 群聊绑定后免 `@` 连续对话
