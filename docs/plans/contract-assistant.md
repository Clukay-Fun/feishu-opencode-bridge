# 律所业务管理系统方案

## 定位

面向法律从业者（律师 / 法务）的业务管理系统，通过飞书 Bot + 多维表格实现：
- 合同起草（模板化）+ 合同信息提取 → 合同台账
- 付款计划跟踪 + 到期提醒
- 发票识别 + 开票提醒
- 案件管理 + 阶段流转提醒

文件通过飞书云文档/网盘传递，结构化数据存 Bitable，表间通过关联字段串联。

## Skill 总览

| Skill | 触发指令 | 输入 | 输出 |
|-------|---------|------|------|
| `contract-draft` | `/起草合同` | 合同类型 + 关键参数 | 合同文档（飞书文档）+ 写入台账 |
| `contract-extract` | `/合同录入` | 合同文件（云文档链接） | 提取关键信息写入台账 + 自动生成付款计划 |

| `reminder-service` | 自动（cron） | — | 扫描到期项，推送提醒卡片 |

## Skill 1：contract-draft（合同起草）

### 触发

```
@bot /起草合同 技术服务合同 甲方XX科技 金额28万 分三期付款
```

参数不全时多轮引导补齐。

### 流程

```
用户提供合同类型 + 参数
  │
  ▼
[1] 匹配模板
    从 Bitable 模板库检索对应类型
    ├── 有模板 → 模板 + 参数填充
    └── 无模板 → AI 从零生成
  │
  ▼
[2] AI 生成草稿
    基于模板 + 参数 + 法律要求生成完整合同
    自动补充必要条款（争议解决、不可抗力…）
  │
  ▼
[3] 创建飞书文档
    输出到用户指定的文档空间
  │
  ▼
[4] 写入合同台账
    同时根据付款条款自动生成付款计划行
  │
  ▼
[5] 输出摘要卡片
```

### 模板库 Bitable 结构

| 字段 | 类型 | 说明 |
|------|------|------|
| 合同类型 | 单选 | 劳动、技术服务、租赁、买卖… |
| 模板名称 | 文本 | 如「标准劳动合同 v2」 |
| 模板内容 | 文本 | Markdown，`{{变量}}` 占位 |
| 必填参数 | 文本 | JSON：`["甲方","乙方","金额","期限"]` |
| 更新时间 | 日期 | |

### 输出卡片示例

```
📝 合同草稿已生成

类型：技术服务合同
模板：标准技术服务合同 v3

关键条款：
  甲方：XX 科技有限公司
  乙方：YY 律师事务所
  金额：¥280,000
  期限：2026-04-12 至 2027-04-11
  付款：分三期（30% / 40% / 30%）

已写入合同台账，付款计划已生成（3 期）

[打开飞书文档编辑]  [查看合同台账]

⚠️ 请仔细核对所有条款后再签署。
```

## Skill 2：contract-extract（合同录入）

### 触发

```
@bot /合同录入
```

附带飞书云文档链接或网盘文件链接。

### 流程

```
用户发送合同文件链接
  │
  ▼
[1] 通过飞书 API 获取文档内容
    云文档 → docx.content API
    网盘文件 → 下载后本地解析
  │
  ▼
[2] AI 提取关键信息
    合同编号、甲乙方、金额、期限、付款方式…
  │
  ▼
[3] 写入合同台账
  │
  ▼
[4] 解析付款条款，自动生成付款计划
    「分三期：30% 预付 + 40% 中期验收 + 30% 终验」
    → 生成 3 行付款计划，应付日期根据合同期限推算
  │
  ▼
[5] 输出摘要卡片 + 确认
    用户可修正 AI 提取有误的字段
```

## Skill 3：invoice-recognize（发票识别）

### 触发

```
@bot /识别发票
```

附带发票照片或 PDF。

### 流程

```
用户发送发票图片/PDF
  │
  ▼
[1] AI 视觉识别（多模态）
    提取：发票号、类型、开票方、收票方、
    金额、税额、税号、开票日期
  │
  ▼
[2] 智能关联
    根据金额 + 开票方/收票方匹配合同台账
    找到对应的付款计划期次
    ├── 唯一匹配 → 自动关联
    ├── 多个候选 → 列出让用户选择
    └── 无匹配 → 写入发票表，不关联
  │
  ▼
[3] 写入发票记录表
    更新对应付款期的发票状态
  │
  ▼
[4] 输出确认卡片
```

### 识别 Prompt 示例

```
识别以下发票图片中的信息，以 JSON 格式输出：
{
  "invoice_no": "发票号码",
  "invoice_type": "增值税专用发票|增值税普通发票|电子发票",

  "receiver": "收票方名称",
  "amount_excl_tax": 数字,
  "tax_amount": 数字,
  "total_amount": 数字,
  "issue_date": "2026-04-10",
  "tax_id": "纳税人识别号"
}
```

### 输出卡片示例

```

发票号：01234567
类型：增值税专用发票
开票方：XX 科技有限公司
```


## Skill 5：reminder-service（提醒服务）

### 机制

bridge 内部 cron 定时任务，每天早上 9:00 扫描 Bitable：

```
每日 09:00
  │
  ▼
[1] 扫描付款计划表
    筛选：状态=待付款 AND 应付日期在提醒窗口内
  │
  ▼
[2] 扫描发票记录表
    筛选：状态=待开票 AND 关联付款期的应付日期临近
  │
  ▼
[3] 扫描案件管理表
    筛选：状态=进行中 AND 下一节点日期在提醒窗口内
  │
  ▼
[4] 汇总推送
    按紧急程度排序，合并为一张提醒卡片
    发送给指定用户/群
```

### 提醒卡片示例

```
🔔 今日待办提醒（2026-04-12）

── 付款 ──────────────────────────────
⏰ 技术服务合同（HT-2026-0042）· 第 1 期
   应收 ¥84,000 · 应付日期 2026-04-15（3 天后）
   发票状态：已收票

⏰ 租赁合同（ZL-2026-0018）· 第 2 期
   应付 ¥32,000 · 应付日期 2026-04-12（今天）
   发票状态：待开票 ← 需要先开票！

── 开票 ──────────────────────────────
📋 委托代理合同（DL-2026-0003）· 第 1 期
   需开票 ¥50,000 · 付款日期 2026-04-19（7 天后）

── 案件 ──────────────────────────────

   截止 2026-04-15（3 天后）
```

### 提醒配置

提醒窗口可在每个条目的「提醒设置」字段配置：
- 提前 3 天
- 提前 7 天
- 仅当天
- 不提醒

逾期未处理的项目每天重复提醒，直到状态更新。

## 技术实现要点

### 文件获取

通过飞书云文档/网盘 API 获取文件内容，不依赖消息附件：

```
用户发送云文档链接
  → 解析链接获取 document_id / file_token
  → 调用飞书 API 获取内容
    云文档：GET /open-apis/docx/v1/documents/{id}/raw_content
    网盘：GET /open-apis/drive/v1/files/{token}/download
```

### Cron 实现

在 bridge 的 `BridgeApp.start()` 中注册定时任务：

```typescript
// 使用 setInterval 实现简单 cron
const SCAN_INTERVAL = 24 * 60 * 60 * 1000; // 24h
setInterval(() => reminderService.scan(), SCAN_INTERVAL);
```

首次启动时计算距离下一个 09:00 的延迟，之后每 24h 执行一次。

### Bitable 操作

统一封装 Bitable CRUD：

```typescript
interface BitableClient {
  // 查询记录（支持筛选）
  listRecords(appToken: string, tableId: string, filter?: string): Promise<Record[]>;
  // 创建记录
  createRecord(appToken: string, tableId: string, fields: Record<string, any>): Promise<string>;
  // 更新记录
  updateRecord(appToken: string, tableId: string, recordId: string, fields: Record<string, any>): Promise<void>;
}
```

### 发票识别

使用多模态 AI（视觉能力）识别发票图片：
- 用户发送图片 → 飞书消息 API 获取图片 → 送入多模态模型
- 或用户发送 PDF → 解析首页为图片 → 送入多模态模型

## 与知识库联动

```
contract-draft
  │
  └── 生成条款时调用 knowledge-query
      检索相关法条，确保条款合规

case-manage
  │
  └── 录入案件时调用 knowledge-query
      检索类似案例和适用法条
```

## 不做的事

- 不做合同审查（后续迭代）
- 不做合同对比（后续迭代）
- 不做电子签章（需 CA 资质）
- 不做合同审批流（用飞书原生审批）
- 不做 OCR 扫描件合同识别（初期只支持可选中文本的文件；发票识别除外，发票用多模态 AI）
- 不做多租户隔离（单人/信任小组定位）
- 不做财务对账（只做提醒，不做资金流管理）

## 实施顺序

1. **Phase 1**：Bitable 表结构创建 + BitableClient 封装
2. **Phase 2**：contract-extract — 合同录入 + 自动生成付款计划，验证核心链路
3. **Phase 3**：invoice-recognize — 发票识别 + 智能关联付款期
4. **Phase 4**：reminder-service — 定时扫描 + 提醒推送
5. **Phase 5**：contract-draft — 模板起草 + 写入台账
6. **Phase 6**：case-manage — 案件管理 + 阶段提醒
