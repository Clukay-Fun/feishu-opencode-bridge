# 法律知识库方案

## 定位

面向法律从业者的知识库系统，支持：
- 批量导入法律文档（PDF/Word），AI 自动提取问答对
- 用户提问时，基于 embedding 语义检索，返回多条带来源引用的答案

实现为两个 lark-cli workflow skill：`knowledge-ingest`（入库）和 `knowledge-query`（检索）。

## 多维表格结构

在飞书 Bitable 中创建知识库表：

| 字段 | 类型 | 说明 |
|------|------|------|
| 问题 | 文本 | AI 提取的问题 |
| 答案 | 文本 | AI 提取的答案 |
| 标签 | 多选 | 法律领域分类（合同、劳动、知产…） |
| 法条 | 文本 | 关联法条引用（如《民法典》第 585 条） |
| 源文件 | 文本 | 原始文档名称 |
| 页码/章节 | 文本 | 在原文档中的位置 |
| embedding | 文本 | 问题+答案的向量（JSON 序列化） |
| 入库时间 | 日期 | 自动填充 |

## Skill 1：knowledge-ingest（入库）

### 触发方式

```
@bot /知识入库
```

附带飞书文档链接或上传 PDF/Word 文件。

### 流程

```
用户上传文档
  │
  ▼
[1] 文档解析
    ├── PDF → 按页提取文本，保留页码
    ├── Word → 按段落/标题提取，保留章节
    └── 飞书文档 → 通过 API 获取结构化内容
  │
  ▼
[2] 分块
    按语义段落切分，每块 500-1000 字
    保留元信息：源文件名、页码/章节号
  │
  ▼
[3] AI 提取问答对
    对每个文本块，调用 LLM 提取：
    - 问题（用户可能怎么问）
    - 答案（基于原文的准确回答）
    - 标签（法律领域分类）
    - 法条（如有明确引用）
  │
  ▼
[4] 生成 embedding
    对「问题 + 答案」拼接文本生成向量
  │
  ▼
[5] 写入 Bitable
    每条问答对写入一行，包含所有字段
```

### AI 提取 Prompt 示例

```

4. 一个片段可提取多条问答对
5. 标签从以下选择：合同、劳动、知产、公司、侵权、婚姻家事、刑事、行政、诉讼程序、其他

源文件：{filename}

文本片段：
---
{chunk}

以 JSON 数组格式输出：
[
  {
    "question": "...",
    "answer": "...",
```

## Skill 2：knowledge-query（检索）

### 触发方式

```
@bot /法律咨询 员工试用期最长多久？
```

或在对话中直接提问，由 bridge 路由判断是否命中知识库。

### 流程

```
用户提问
  │
  ▼
[1] 生成查询 embedding
    对用户问题生成向量
  │
  ▼
[2] 语义检索
    与 Bitable 中所有条目的 embedding 计算余弦相似度
    取 top-K（K=10）候选
  │
  ▼
[3] AI 重排序
    将 top-K 的问题+答案送入 LLM 重排序
    过滤无关项，按相关性排序
    保留 top-N（N=3~5）
  │
  ▼
[4] 格式化输出
    每条答案附带来源引用（源文件 + 页码）
    同一问题的多个答案全部列出
```


### Embedding 方案

```typescript
// 余弦相似度
function cosineSimilarity(a: number[], b: number[]): number {
  let dot = 0, normA = 0, normB = 0;
  for (let i = 0; i < a.length; i++) {
    dot += a[i] * b[i];
    normA += a[i] * a[i];
    normB += b[i] * b[i];
  }
  return dot / (Math.sqrt(normA) * Math.sqrt(normB));
}
```

### Embedding 存储

两种方案，按规模选择：

**方案 A：Bitable 内嵌（推荐，< 5000 条）**
- embedding 序列化为 JSON 字符串存在 Bitable 字段中
- 查询时拉取所有条目，本地计算相似度
- 优点：零额外依赖，数据集中在飞书
- 缺点：条目多时拉取慢

**方案 B：本地 SQLite + 向量索引（> 5000 条）**
- 本地 SQLite 存储 embedding，Bitable 只存业务字段
- 使用 `sqlite-vss` 或暴力扫描
- 优点：检索快
- 缺点：需要同步 Bitable ↔ SQLite

初期推荐方案 A，数据量增长后迁移到方案 B。

### 文档解析

| 格式 | 解析方式 | 依赖 |
|------|---------|------|
| PDF | `pdf-parse` npm 包 | 需安装 |
| Word (.docx) | `mammoth` npm 包 | 需安装 |
| 飞书文档 | 飞书开放 API `docx.content` | lark-cli |
| 纯文本 | 直接读取 | 无 |

### 分块策略

```
文档文本
  │
  ▼
按页/章节初步切分
  │
  ▼
按段落边界二次切分
  │
  ▼
每块 500-1000 字（中文字符计数）
重叠窗口 100 字（避免截断关键信息）
  │
  ▼
附带元信息：{ filename, page, section }
```

## 不做的事

- 不做实时文档同步（入库是一次性操作，更新需重新入库）
- 不做权限控制（当前单人/信任小组定位）
- 不做答案生成（只检索已入库的问答对，不基于原文重新生成）
- 不做多语言支持（仅中文）
- 不替代律师专业意见（卡片底部加免责声明）

## 免责声明

检索结果卡片底部固定附带：

```
⚠️ 以上内容仅供参考，不构成法律意见。具体问题请咨询专业律师。
```

## 实施顺序

1. **Phase 1**：knowledge-query 骨架 + Bitable 手动录入几条测试数据 → 验证检索链路
2. **Phase 2**：knowledge-ingest 文档解析 + AI 提取 → 自动化入库
3. **Phase 3**：优化检索质量（重排序、同义词扩展、多轮追问）
