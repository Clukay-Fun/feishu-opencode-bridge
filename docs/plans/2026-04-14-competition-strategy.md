# 参赛方针整理

## 当前目标

围绕劳动法业务，整理一条清晰、可落地、可录屏的参赛主线。

本次作品不再展示一组分散功能，而是强调一条完整的业务闭环，让评委看到系统对律所真实办案流程的支撑能力。

## 叙事主线

建议统一为：

**律所劳动法业务的 AI 全流程支撑：从知识入库到案件办理**

用一个完整的劳动争议案件串起主要能力：

```text
律所接到一个劳动争议案件
  │
  ├─ [1] 知识准备
  │      将劳动法相关法规、书籍、资料入库
  │
  ├─ [2] 快速研究
  │      提问“经济补偿怎么算”等问题
  │      返回带法条依据的答案
  │
  ├─ [3] 劳动 skill · 证据链分析
  │      对本地或飞书云盘中的多份案件材料进行汇总分析
  │      生成结构化证据清单、事实时间线与风险提示
  │
  ├─ [4] 劳动 skill · 基于证据链的文书起草
  │      基于案件事实、争议焦点与证据链
  │      起草仲裁申请书、答辩状、情况说明等文书
  │
  ├─ [5] 案件管理
  │      录入案件信息、设置节点提醒、生成工作记录
  │
  └─ [6] 结案归档
         归档材料、统计工时、补充发票提醒
```

## 展示原则

评分上，技术方案与法律专业质量是主体，必须深展示。

业务广度只需要证明系统完整，不需要每个功能都展开。

因此本次展示遵循以下原则：

- 选一条主线，把其他能力挂在主线之上
- 深挖知识库检索与法律专业输出质量
- 重点补齐证据整理，作为案件实务场景中的高频能力
- 不把“合同审查”当成孤立功能，而是把它收敛为证据链分析中的关键文书专项分析
- 文书起草建立在证据链分析之上，避免直接从原始材料跳到草稿
- 新法新规梳理暂不做现场重点演示，优先写入“优化方向”

## 飞书使用原则

本次 AI 演示主题要求放在飞书内完成，因此录屏应尽量统一在飞书环境中进行。

展示重点不是“飞书界面本身”，而是“律师在飞书内完成真实工作流”。

因此系统定位应当是：

- 飞书是主要交互界面
- OpenCode 是背后的 AI 运行能力
- `lark-cli` 负责把 AI 结果真正写回飞书文档、画板、表格等载体

## 录屏总策略

录屏全程尽量在飞书中操作，减少工具切换。

推荐节奏如下：

| 时间 | 环节 | 展示深度 |
|------|------|----------|
| 0-1 分钟 | 场景引入，快速扫过群聊、文档、表格等整体能力 | 快 |
| 1-4 分钟 | 知识库入库与检索 | 深 |
| 4-6 分钟 | 劳动 skill：证据链分析，生成案件工作文档 | 深 |
| 6-7 分钟 | 劳动 skill：基于证据链的文书起草 | 中 |
| 7-8 分钟 | 案件管理、提醒、工作记录 | 快 |
| 8-9 分钟 | 结案归档、工时统计、发票提醒 | 快 |
| 9-10 分钟 | 律师总结、人机分工、优化方向 | 收尾 |

## 功能取舍

### 必须重点做好的能力

#### 1. 知识库入库与检索

这是展示技术方案与法律专业质量的主力模块。

需要体现：

- 劳动法素材可被稳定入库
- 检索问题贴近劳动法实务
- 输出中带法律依据、条文引用或规则说明
- 回答不是泛泛而谈，而是能用于办案辅助

#### 2. 劳动 skill · 证据链分析

这是当前需要补做、但非常值得做的能力。

相比单独审查一份合同，证据链分析更贴近律师高频工作，也更容易体现“案件办理全流程”。

建议定义为：

- 输入：本地目录或飞书云盘中的多份案件材料
- 处理方式：先汇总材料内容，再由 AI 基于整体上下文输出完整证据链分析
- 输出载体：飞书文档

飞书文档内建议包含：

- 案件与整理范围说明
- 证据清单表格
- 时间线
- 争议焦点与风险提示
- 待补材料与下一步建议
- 流程图或证据关系图

### 可选展示能力

#### 3. 合同与关键文书专项审查

合同审查本质上是证据链分析中的一个专项子能力。

如果时间充裕，可作为证据链分析后的延伸动作，展示 AI 对劳动合同、离职协议、通知书等关键材料中的风险条款、试用期、解除条件等内容进行识别与提示。

如果时间不足，不必单独扩成完整模块。

#### 4. 基于证据链的文书起草

文书起草可做，但必须收敛。

最稳妥的方式是只选一个劳动法场景，例如：

- 劳动争议仲裁申请书
- 劳动争议答辩状

前提是模板和提示词需要经过律师审核，避免 AI 自由生成导致法律专业质量不稳定。

更合理的链路应当是：

```text
案件材料
  -> 证据链分析
  -> 结构化案件事实
  -> 文书起草
```

### 放入优化方向的能力

#### 5. 新法新规智能梳理

这个方向适合写入报告中的“优化方向”。

当前表述建议为：

系统未来可持续跟踪劳动法领域新发布的法律法规、司法解释及部门规章，自动识别与在办案件的关联性，并生成影响摘要与合规建议。

如果现有系统已经能支撑“法规入库后检索出影响点”，可在录屏中点到为止；否则不作为核心演示内容。

## 劳动 skill 的产品定义

当前更推荐采用整体分析思路，而不是逐文件孤立分析。

目标形态：

- 用户指定一个本地目录或飞书云盘文件夹
- 系统读取其中的劳动合同、工资单、考勤记录、聊天记录、解除通知等材料
- AI 从全局视角整理证据链、争议焦点与缺口
- 最终输出到飞书文档

建议的输出结构如下：

| 序号 | 证据名称 | 证据类型 | 证明事项 | 备注 |
|------|----------|----------|----------|------|
| 1 | 劳动合同.pdf | 书证 | 证明双方存在劳动关系、岗位与期限约定 | 含试用期条款 |
| 2 | 工资流水.pdf | 书证 | 证明工资标准与发放情况 | 可用于补偿计算 |
| 3 | 解除通知.docx | 书证 | 证明单位单方解除劳动合同 | 需核对送达时间 |
| 4 | 考勤记录.xlsx | 书证 | 证明劳动者出勤情况 | 可反驳旷工主张 |

同时可附加三部分简述：

- 证据链概览
- 关键事实时间线
- 尚需补充的关键证据提示

## 飞书文档落地方式

结合当前能力，飞书文档可以作为证据整理结果的最终载体。

目前可行的实现方式是：

- 使用 `lark-cli docs +create` 或 `lark-cli docs +update` 写入文档正文
- 用 Markdown 表格或 `<lark-table>` 生成证据清单
- 插入空白 whiteboard 后，通过 `lark-cli docs +whiteboard-update` 写入 Mermaid 流程图或关系图

因此“劳动 skill 的证据链分析 -> 飞书文档输出”是当前最适合比赛演示的落地方向。

## 报告写法建议

报告可继续沿用五个模块，但叙事方式改成案件生命周期。

### 1. 团队介绍

控制在约 150 字，简述成员背景与分工。

### 2. 作品简介

控制在约 200 字，统一描述为：

基于飞书的律所 AI 全流程管理系统，覆盖知识管理、劳动证据链分析、基于证据链的文书起草、案件办理、工时统计等核心业务。

### 3. 选题背景

控制在约 350 字，聚焦劳动法业务中的真实痛点：

- 知识材料分散
- 重复研究成本高
- 证据链分析与文书起草耗时
- 办案经验难沉淀复用

### 4. 解决方案

控制在约 800 字，按案件生命周期展开，说明每个环节 AI 做了什么。

技术层面重点点到以下关键词：

- Agent 自主执行
- 多模型路由
- embedding 语义检索
- Skill 注入
- 飞书文档与工作流集成

### 5. 实际效果与优化方向

控制在约 500 字。

先写当前可量化的成效，再写后续优化方向，例如新法新规智能梳理。

## 当前执行优先级

剩余时间有限，最优先的三件事是：

1. 确定一个具体劳动争议案件故事线，作为整场录屏的统一案例
2. 准备好演示数据，尤其是劳动法知识库素材和证据材料目录
3. 补齐劳动 skill 的证据链分析到飞书文档的最小可演示版本

## 推荐分工

- 一人负责录屏操作，保证流程稳定
- 一人负责律师视角旁白，强调法律专业判断与人机分工
- 一人负责演示数据、模板、提示词与结果校验

## 后续实现建议

如果只做一个最小可演示闭环，优先顺序建议如下：

1. 知识库入库与检索稳定可演示
2. 证据链分析输出飞书文档
3. 基于证据链的文书起草补齐一个稳定场景
4. 案件管理与提醒做快速展示
5. 新法新规梳理写入优化方向

## 一句话总结

这次参赛作品应当从“功能合集”切换为“劳动争议案件全流程 AI 辅助办案系统”，并以飞书作为统一演示入口，以知识库检索和劳动 skill 的证据链分析作为核心亮点。
