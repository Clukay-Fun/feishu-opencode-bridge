# 律所业务管理系统方案

## 定位

面向法律从业者（律师 / 法务）的业务管理系统，通过飞书 Bot + 多维表格实现：
- 合同起草（模板化）+ 合同信息提取 → 合同台账
- 合同台账内付款节点、收款与开票状态跟踪
- 发票识别 + 开票提醒
- 案件管理 + 阶段流转提醒

文件通过飞书云文档/网盘传递，结构化数据存 Bitable，表间通过关联字段串联。

## 数据模型（Bitable 表结构）

```
合同台账 ──1:N──→ 发票记录

```

## 能力分层

当前方案里需要区分三层概念，避免把领域入口、专项能力和底层模块混在一起：

### 1. 领域 skill

领域 skill 面向最终用户，是用户进入某一业务语境后的总入口。

#### labor-skill

`labor-skill` 是劳动争议场景的领域 skill，不应和 `contract-draft` 这类专项能力并排理解。

它的职责是：

- 接收劳动争议相关材料与问题
- 调用劳动法知识库
- 做证据链分析
- 生成案件工作台文档
- 在需要时进一步起草劳动争议文书

建议内部 workflow：

```text
labor-skill
  ├─ analyze-evidence-chain
  ├─ generate-labor-workbench
  ├─ query-labor-knowledge
  └─ draft-labor-document
```

调用关系：

```text
用户进入 labor-skill
  -> 上传材料 / 补充背景
  -> labor-skill 内部调用证据分析、知识库、文档生成等底层能力
  -> 输出劳动案件工作台或劳动文书
```

说明：

- `labor-skill` 不等于 `contract-draft`
- `labor-skill` 也不要求用户自己挑底层模块
- 它是一个面向劳动争议场景的编排入口

### 2. 独立专项 skill

独立专项 skill 面向单一任务，可在多个领域 skill 中复用，也可以独立提供给用户。

| Skill | 触发指令 | 输入 | 输出 | 定位 |
|-------|---------|------|------|------|
| `contract-draft` | `/起草合同` | 合同类型 + 关键参数 | 合同文档（飞书文档）+ 写入台账 | 通用合同起草专项 skill |

说明：

- `contract-draft` 是独立专项 skill
- 它可以被 `labor-skill` 调用，但不属于 `labor-skill` 的核心主干
- 只有在劳动场景明确需要起草劳动合同、竞业协议、离职协议、offer 等文书时，`labor-skill` 才可能转调 `contract-draft`

### 3. 底层能力模块

底层能力模块是面向系统实现的能力，不一定直接暴露为最终用户的一级入口。

| 模块 | 当前文档中的旧命名 | 主要职责 |
|------|------------------|----------|
| 合同信息提取 | `contract-extract` | 读取合同文件并写入合同台账 |
| 发票识别 | `invoice-recognize` | 识别发票并关联合同 |
| 案件管理 | `case-manage` | 写入案件表、推进阶段、维护期限 |
| 提醒服务 | `reminder-service` | 定时扫描台账并推送提醒 |
| 证据链分析 | 新增 | 读取多材料、提取事实、识别风险和缺口 |
| 工作台生成 | 新增 | 输出飞书文档、画板、多维表格入口 |
| 知识库查询 | 新增 | 提供法条、争议点和规则支撑 |
| 文书起草 | 新增 | 基于事实与证据链起草文书 |

建议理解为：

```text
底层能力模块
  -> 被独立专项 skill 调用
  -> 也被领域 skill 编排调用
```

### 总体关系

推荐统一成下面这个结构：

```text
领域 skill
  └─ labor-skill

独立专项 skill
  └─ contract-draft

底层能力模块
  ├─ contract-extract
  ├─ invoice-recognize
  ├─ case-manage
  ├─ reminder-service
  ├─ analyze-evidence-chain
  ├─ generate-workbench
  ├─ knowledge-query
  └─ draft-document
```

  ├─ evidence-extract
  ├─ timeline-build
  ├─ workbench-generate
  ├─ document-draft
  ├─ contract-draft
  ├─ contract-extract
  ├─ invoice-recognize

  └─ reminder-service
```

## Skill 1：contract-draft（合同起草）

### 触发

```
@bot /起草合同 技术服务合同 甲方XX科技 金额28万 分三期付款
```

参数不全时多轮引导补齐。

### 流程

```
用户提供合同类型 + 参数
  │
  ▼
[1] 匹配模板
    从 Bitable 模板库检索对应类型
    ├── 有模板 → 模板 + 参数填充
    └── 无模板 → AI 从零生成
  │
  ▼
[2] AI 生成草稿
    基于模板 + 参数 + 法律要求生成完整合同
    自动补充必要条款（争议解决、不可抗力…）
  │
  ▼
[3] 创建飞书文档
    输出到用户指定的文档空间
  │
  ▼
[4] 写入合同台账
    付款安排直接写入 `付款节点`
  │
  ▼
[5] 输出摘要卡片
```

### 模板库 Bitable 结构

| 字段 | 类型 | 说明 |
|------|------|------|
| 合同类型 | 单选 | 劳动、技术服务、租赁、买卖… |
| 模板名称 | 文本 | 如「标准劳动合同 v2」 |
| 模板内容 | 文本 | Markdown，`{{变量}}` 占位 |
| 必填参数 | 文本 | JSON：`["甲方","乙方","金额","期限"]` |
| 更新时间 | 日期 | |

### 输出卡片示例

```
📝 合同草稿已生成

类型：技术服务合同
模板：标准技术服务合同 v3

关键条款：
  甲方：XX 科技有限公司
  乙方：YY 律师事务所
  金额：¥280,000
  期限：2026-04-12 至 2027-04-11
  付款：分三期（30% / 40% / 30%）

已写入合同台账
付款节点已整理进合同字段

[打开飞书文档编辑]  [查看合同台账]

⚠️ 请仔细核对所有条款后再签署。
```

## Skill 3：invoice-recognize（发票识别）

### 触发

```
@bot /识别发票
```

附带发票照片或 PDF。

### 流程

```
用户发送发票图片/PDF
  │
  ▼
[1] AI 视觉识别（多模态）
    提取：发票号、类型、开票方、收票方、
    金额、税额、税号、开票日期
  │
  ▼
[2] 智能关联
    根据合同号、付款方、金额、开票日期匹配合同台账
    ├── 唯一匹配 → 自动关联
    ├── 多个候选 → 列出让用户选择
    └── 无匹配 → 写入发票表，不关联
  │
  ▼
[3] 写入发票记录表
    回写合同台账的发票附件或开票状态相关字段
  │
  ▼
[4] 输出确认卡片
```

### 识别 Prompt 示例

```
识别以下发票图片中的信息，以 JSON 格式输出：
{
  "invoice_no": "发票号码",
  "invoice_type": "增值税专用发票|增值税普通发票|电子发票",
  "issuer": "开票方名称",
  "receiver": "收票方名称",
  "amount_excl_tax": 数字,
  "tax_amount": 数字,
  "total_amount": 数字,
  "issue_date": "2026-04-10",
  "tax_id": "纳税人识别号"
}
```

### 输出卡片示例

```
🧾 发票识别完成

发票号：01234567
类型：增值税专用发票
开票方：XX 科技有限公司
金额：¥84,000（税额 ¥5,040）
开票日期：2026-04-10

已关联：技术服务合同（HT-2026-0042）

## Skill 5：reminder-service（提醒服务）

### 机制

bridge 内部 cron 定时任务，每天早上 9:00 扫描 Bitable：

```
每日 09:00
  │
  ▼
[1] 扫描合同台账
    筛选：存在未收款 / 未开票金额 / 临近关键付款节点
  │
  ▼
[2] 扫描发票记录表
    筛选：待归档、缺少关联合同或需要补核验
  │
  ▼
[3] 扫描案件管理表
    筛选：状态=进行中 AND 下一节点日期在提醒窗口内
  │
  ▼
[4] 汇总推送
    按紧急程度排序，合并为一张提醒卡片
    发送给指定用户/群
```

### 提醒卡片示例

```
🔔 今日待办提醒（2026-04-12）

── 合同 ──────────────────────────────
⏰ 技术服务合同（HT-2026-0042）
   未收款 ¥84,000
   付款节点：合同签署后 15 日内支付 30%

⏰ 租赁合同（ZL-2026-0018）
   未开票金额 ¥32,000
   当前需要补开发票

── 开票 ──────────────────────────────
📋 委托代理合同（DL-2026-0003）
   需核验发票 ¥50,000 · 开票日期 2026-04-19

── 案件 ──────────────────────────────
⚖️ XX 科技 vs YY 公司 · 举证期限
   截止 2026-04-15（3 天后）
   当前阶段：举证

[查看全部待办]
```

### 提醒配置

提醒优先基于：

- 合同台账中的未收款、未开票金额、付款节点
- 发票记录中的开票日期、关联合同和附件状态
- 案件管理中的关键期限字段

逾期未处理的项目可每天重复提醒，直到状态更新。

## 技术实现要点

### 文件获取

通过飞书云文档/网盘 API 获取文件内容，不依赖消息附件：

```
用户发送云文档链接
  → 解析链接获取 document_id / file_token
  → 调用飞书 API 获取内容
    云文档：GET /open-apis/docx/v1/documents/{id}/raw_content
    网盘：GET /open-apis/drive/v1/files/{token}/download
```

### Cron 实现

在 bridge 的 `BridgeApp.start()` 中注册定时任务：

```typescript
// 使用 setInterval 实现简单 cron
const SCAN_INTERVAL = 24 * 60 * 60 * 1000; // 24h
setInterval(() => reminderService.scan(), SCAN_INTERVAL);
```

首次启动时计算距离下一个 09:00 的延迟，之后每 24h 执行一次。

### Bitable 操作

统一封装 Bitable CRUD：

```typescript
interface BitableClient {
  // 查询记录（支持筛选）
  listRecords(appToken: string, tableId: string, filter?: string): Promise<Record[]>;
  // 创建记录
  createRecord(appToken: string, tableId: string, fields: Record<string, any>): Promise<string>;
  // 更新记录
  updateRecord(appToken: string, tableId: string, recordId: string, fields: Record<string, any>): Promise<void>;
}
```

### 发票识别

使用多模态 AI（视觉能力）识别发票图片：
- 用户发送图片 → 飞书消息 API 获取图片 → 送入多模态模型
- 或用户发送 PDF → 解析首页为图片 → 送入多模态模型

## 不做的事

- 不做合同审查（后续迭代）
- 不做合同对比（后续迭代）
- 不做电子签章（需 CA 资质）
- 不做合同审批流（用飞书原生审批）
- 不做 OCR 扫描件合同识别（初期只支持可选中文本的文件；发票识别除外，发票用多模态 AI）
- 不做多租户隔离（单人/信任小组定位）
- 不做财务对账（只做提醒，不做资金流管理）

## 劳动 Skill 工作台增强

### 定位调整

劳动 skill 的输出不应停留在一篇分析报告，而应升级为一个可继续协作、补证、追踪和起草的飞书工作台。

建议拆成 5 个承载层：

- 飞书文档：承载结论、总览、结构化阅读和对外展示
- 多维表格：承载证据台账、状态追踪、缺口管理和画册展示
- 画板：承载时间线、证据关系图、争议焦点图
- 思维导图：承载请求项拆解、案件事实树
- 流程图：承载补证路径、仲裁路径、内部起草流程

核心原则：

```text
文档负责读
多维表格负责管
画板负责看
思维导图负责理
流程图负责走
```

### 推荐输出结构

劳动 skill 的增强版输出建议统一为：

| 业务对象 | 飞书载体 | 主要用途 |
|----------|----------|----------|
| Case | 飞书文档 | 案件总览页、结论摘要、协作文档 |
| Material / Evidence Item | 多维表格 | 证据台账、文件挂载、状态管理 |
| Facts / Issues | 画板 | 时间线、证据关系图、争议焦点图 |
| Claims | 思维导图 | 请求项与支撑证据树 |
| Next Actions | 流程图 | 补证路径、仲裁路径、行动建议 |

### 飞书文档版式建议

首页建议使用 `分栏 + 高亮块`，而不是直接进入正文。

推荐布局：

```text
左栏：
  案件名称
  当前阶段
  当前结论
  风险等级

右栏：
  当前缺口
  待补材料
  下一步动作
  责任人 / 更新时间
```

推荐块类型：

- 红色高亮块：关键证据缺口
- 黄色高亮块：待核实事实
- 蓝色高亮块：建议下一步
- 引述块：当前结论摘要

正文建议保持稳定结构：

1. 案件摘要
2. 核心判断
3. 证据链总表
4. 关键事实时间线
5. 争议焦点与风险
6. 法律依据与知识库支撑
7. 待补材料与下一步建议

### 多维表格建议

建议新增 `证据台账` 表，至少包含：

| 字段 | 类型 | 说明 |
|------|------|------|
| 证据名称 | 文本 | 如“劳动合同首页”“2023-02-22 解除通知” |
| 证据类型 | 单选 | 合同、工资单、考勤、聊天记录、录音转写、通知书等 |
| 材料来源 | 文本 | 用户上传、邮件导出、聊天截图等 |
| 证据日期 | 日期 | 材料或事件日期 |
| 证明事实 | 长文本 | 支持什么事实 |
| 对应争议点 | 多选/关联 | 劳动关系、工资、加班、解除、补偿等 |
| 支持方向 | 单选 | 支持劳动者、支持用人单位、中性 |
| 证明力 | 单选 | 强、中、弱 |
| 当前状态 | 单选 | 已收集、待补原件、待核验、已使用 |
| 原始文件链接 | 附件/链接 | 对应飞书文件、截图、云文档 |
| 是否关键证据 | 勾选 | 是否属于闭环关键节点 |
| 是否存在缺口 | 勾选 | 是否仍有明显证明缺口 |
| 缺口说明 | 长文本 | 需要补什么 |
| 备注 | 长文本 | 其他说明 |

证据链输出默认增加一层 `证据脱敏`：

- 飞书文档正文默认展示脱敏后的证据摘要
- 证据台账中的姓名、联系方式、身份证号、银行卡号、邮箱、住址等字段默认脱敏展示
- 原始文件仍通过附件或原始文件链接保留，不在正文和摘要表中直接展开
- 时间线、证据关系图、请求项树中若引用个人敏感信息，也统一使用脱敏后的别名或局部遮罩

建议默认规则：

- 姓名：保留姓氏，后续字符打星
- 手机号：`138****5678`
- 身份证号：`310101********1234`
- 银行卡号：仅保留前 4 位和后 4 位
- 邮箱：仅保留首字符和域名
- 详细地址：保留前半段语义，后半段打码

建议至少提供这些视图：

- 全量台账视图
- 关键证据视图
- 缺口证据视图
- 高风险视图
- 图片/截图画册视图

### 可视化建议

默认保留两张画板：

- 时间线
- 证据关系图

按复杂度扩展：

- 请求项思维导图
- 补证与仲裁流程图
- 争议焦点图
- 举证责任图

### 增量实施顺序

建议在原实施顺序后增加一个劳动 skill 增强阶段：

1. **Labor Phase 1**：文档正文 + 时间线画板 + 证据关系图画板
2. **Labor Phase 2**：首页分栏 + 高亮块 + 文档入口区块
3. **Labor Phase 3**：证据台账 Base + gallery/card 视图
4. **Labor Phase 4**：请求项思维导图 + 补证流程图

### 一句话结论

劳动 skill 的增强方向不是“把文档写得更长”，而是把输出升级为案件工作台：

```text
文档做总览页
多维表格做证据台账
画板做案件可视化
思维导图做主张拆解
流程图做行动路线
```
